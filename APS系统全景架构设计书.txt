APS 系统全景架构设计书
一、 系统总体架构总览
1.1 设计理念
本系统遵循“功能模块化、技术微服务化、数据一体化、决策智能化”的设计理念，构建基
于云原生架构的跨厂协同高级计划排程系统。系统旨在打破总装厂与配套厂的信息壁垒，通过
“五大中心”协同，实现从需求接入到排程执行、再到监控反哺的全链路闭环。
1.2 架构核心逻辑
系统架构由业务架构、功能架构、技术架构、数据架构四大支柱构成，形成“三层架构协同、
全链路闭环”体系：
● 业务闭环：需求接入 → 资源匹配 → 规则校验 →协同排程→ 监控反馈 → 优化迭代。
● 跨厂协同：通过加密网关与统一数据标准，实现总装厂与配套厂在库存、产能、计划层面的
实时联动。
● 智能驱动：引入多智能体（Agent）协商机制与LLM（大语言模型）助手，实现排程的自愈
与优化。
二、 业务架构
业务架构以“五大中心”为核心，明确各领域的业务边界与协同关系：
1. 需求中心 (Demand Center)：作为业务输入端，负责多源订单接入、BOM智能递归拆解、
库存水位动态计算及跨厂需求同步，解决“生产什么”的问题。
2. 资源中心 (Resource Center)：作为生产要素底座，通过数字孪生技术对设备、模具、人员、
日历进行全要素建模与实时监控（IoT），解决“用什么生产”的问题。
3. 规则中心 (Rule Center)：作为约束管控机构，管理工艺硬约束与业务软约束，提供仿真沙
箱进行What-If推演，解决“怎么生产才合规”的问题。
4. 协同中心 (Collaboration Center)：作为决策大脑，利用有限能力排程算法与多智能体协商
机制，生成最优排程方案，并处理异常自愈，解决“何时生产”的问题。
5. 监控中心 (Monitoring Center)：作为反馈优化闭环，全链路聚合执行数据，监控“计划vs
执行”偏差，计算KPI并反哺规则参数，解决“生产得怎么样”的问题。
三、 功能架构

功能架构采用分层设计，自下而上分别为基础支撑层、核心业务层、交互展示层。
3.1 交互展示层
提供多终端适配的可视化操作入口，集成AI智能交互能力。
● PC端可视化：
○ 三维卡片墙：展示设备运行状态、模具全生命周期。
○ 穿透式甘特图：支持跨厂联动查看，提供拖拽调序功能。
○ 热力图：展示人员技能矩阵与资源负载分布。
● 移动端应用：扫码报工、异常实时反馈、个人任务看板。
● 智能助手：基于LLM的侧边栏助手，支持自然语言问答与自愈策略推荐。
3.2 核心业务层
1. 需求中心
● 订单管理：多源订单清洗、无效/重复过滤、旺季订单高亮识别、订单版本管理。
● BOM模块：BOM递归拆解、能效批次码生成（强一致性）、父子工单关联树构建。
● 库存模块：跨厂库存接入、库存水位动态计算、替代料智能匹配、齐套性检查。
2. 资源中心
● 设备模块：设备数字孪生模型、IoT实时状态流接入、产能核算、故障监控。
● 模具模块：模具台账、寿命自动扣减与监控、跨厂调拨轨迹跟踪。
● 人员模块：资质校验、技能矩阵管理、跨厂支援工时计算。
● 日历模块：跨厂生产日历配置、工作日历偏差检测与对齐。
3. 规则中心
● 约束规则：硬约束（工艺/设备能力）、软约束（交期/成本评分）配置。
● 参数池：淡/旺/应急多场景参数组配置、参数一键热切换。
● 仿真沙箱：生产快照克隆、What-If假设分析、排程冲突检测与消解建议。
4. 协同中心
● 智能排程：有限能力排程算法（ASAP/JIT）、多方案（成本/交期最优）生成。
● 交互排程：甘特图实时拖拽校验、实时协作锁管理（防止多人冲突）。
● 诊断与自愈：基于Agent的多智能体协商（总装vs配套）、异常根因分析、切厂/外协/局部
重排策略生成。

5. 监控中心
● 数据聚合：MES/IoT报工流处理、全链路数据清洗与汇总。
● 监控看板：全局进度、资源利用率、关键指标（KPI）仪表盘。
● 异常管理：三级异常哨兵、分级告警推送、异常闭环处理流程。
● 反哺优化：KPI计算、规则参数优化建议生成（形成闭环）。
3.3 基础支撑层
● 权限管理：RBAC角色管理、多租户（跨厂）权限控制、操作审计。
● 接口适配：ERP/MES/IoT/WMS接口适配器、数据格式转换。
● 消息服务：通知推送、任务提醒、系统公告。

4.2 微服务拆分与定义
系统后端划分为 8个核心微服务域，实现高内聚低耦合：
1. 需求服务 (Demand/Order)：处理订单清洗、BOM拆解、库存逻辑。
2. 资源服务 (Resource/IoT)：管理资源台账、处理IoT实时流。
3. 规则服务 (Rule/Algorithm)：管理约束库、仿真沙箱快照。
4. 协同服务 (Schedule/Agent)：核心排程引擎、Agent协商框架。
5. 监控服务 (Monitor/KPI)：数据聚合、KPI计算、反哺逻辑。
6. 基础服务 (Auth/Audit)：用户认证、权限校验、日志审计。
7. 移动端服务 (Mobile/Scan)：扫码、现场作业API封装。
8. 通用服务 (OSS/Notify)：文件存储、消息通知服务。
4.3 核心接口规范
遵循 RESTful 规范，部分实时交互采用 WebSocket，所有接口支持数据加密。
● 外部集成：
○ ERP：POST /api/demand/clean (订单同步与清洗)
○ IoT：GET /api/resource/iot/stream (WebSocket获取设备流)
○ MES：POST /api/monitor/feedback (接收报工)
● 内部核心：
○ BOM拆解：POST /api/demand/bom/dissolve (生成跨厂工单树)
○ 拖拽校验：POST /api/schedule/drag/validate (甘特图实时校验)
○ 智能协商：POST /api/schedule/agent/negotiate (触发Agent计算)
○ 自愈重排：POST /api/schedule/heal (执行应急策略)
五、 数据架构
围绕数据全生命周期管控，强化跨厂关联（父子工单、批次码）与状态追踪能力。

5.1 数据分类与核心模型
● 需求数据：
○ 销售订单 (Sales Order)：包含优先级、旺季标识、清洗状态。
○ 生产工单 (Work Order)：核心字段含父工单ID（跨厂关联）、能效批次码（强一致
性）、锁定状态。
● 资源数据：
○ 设备/模具：包含IoT状态、跨厂共享标识、剩余寿命、当前位置。
● 规则数据：
○ 规则定义：类型（硬/软）、权重值、适用场景（淡/旺）。
● 排程数据：
○ 排程结果：方案类型、冲突点JSON数据、甘特图坐标。
● 监控数据：
○ 日志/异常：告警等级、跨厂责任人、闭环反馈内容。
5.2 数据存储策略
采用混合存储模式以优化性能：
● MySQL (关系型)：存储核心业务主数据（订单、BOM、资源台账），主从复制。
● Redis (缓存)：存储高频访问数据（实时库存水位、设备状态、Session）。
● Elasticsearch (搜索)：存储海量日志、异常数据、历史报工，支持快速检索与分析。
● OSS (文件)：存储静态资源、报表文件、现场反馈的图片/语音。
5.3 数据流转逻辑
1. Inbound (入站)：ERP订单 $\rightarrow$ 接口适配 $\rightarrow$ 需求中心（清洗/拆
解） $\rightarrow$ 生产工单库。
2. Internal (内部)：工单需求 + 资源实时状态 + 规则参数 $\rightarrow$ 协同中心（排程/协
商） $\rightarrow$ 排程结果。
3. Outbound (出站)：排程结果 $\rightarrow$ MES执行；执行偏差/报工 $\rightarrow$ 监
控中心 $\rightarrow$ 规则反哺。
4. Cross-Plant (跨厂)：总装厂APS $\longleftrightarrow$ 加密网关 $\longleftrightarrow$
配套厂APS（双向同步与校验）。
六、 安全与运维体系
● 数据安全：全链路HTTPS/MQTT加密传输，跨厂通信双重加密；敏感数据存储加密；严格
的RBAC权限管控与操作日志审计（留存>6个月）。

● 部署架构：基于K8s的容器化部署，支持多环境（测试/预生产/生产）隔离。核心服务（排
程、数据聚合）多实例部署，具备故障自愈能力。
● 运维监控：Prometheus + Grafana实时监控微服务状态、接口调用及系统性能，异常自动
告警。
